# 当十几个 Claude Code 同时 Coding，谁来掌控全局？

> 从四个灵魂拷问到 AI 项目控制塔的设计思考

---

AI coding 的速度已经不再受限于人类敲键盘的速度。但一个新的瓶颈正在浮现——**当 AI 写代码比人类审查代码还快的时候，谁来管项目？**

设想一个场景：你同时维护着四五个项目，每个项目有两三个 Claude Code 在并行推进。有些 Agent 跑在本地，有些跑在云端。它们都在改代码、做决策、提交 PR。而你，作为唯一的人类，怎么知道一切在轨道上？

这不是科幻，这是正在发生的日常。我们围绕这个问题做了一系列设计探索——从最基础的配置规范，到可视化的多项目控制塔平台。这篇文章记录了核心的思考过程。

---

## 一、四个问题

所有设计从一个简单的追问开始：**AI coding 规模化了，人的治理能力怎么规模化？**

拆开来看是四个问题：

**第一，多项目并行时，人怎么掌控全局？** 你不可能盯着十几个终端。你需要的不是在代码层面审查每一个改动，而是在项目层面感知每一个项目的健康状态——哪个在正常推进、哪个卡住了、哪个触发了风险。

**第二，架构质量谁来保证？** 当人类不再深入代码细节，谁来督促 AI 做出合理且安全的架构设计？谁来把握「便于拓展」与「不过度设计」之间的平衡？

**第三，目标如何精准传达？** 人类想要的东西和 AI 理解的东西之间，天然存在一个翻译层。模糊的自然语言描述必然导致跑偏、返工、浪费算力。如何把目标表达成 AI 不会误解的形式？

**第四，如何定义「完成」让 AI 自主迭代？** 当前大多 Agent 框架在每个阶段执行后停下来等人类确认——安全，但低效。如果想让 AI 自己推进到真正完成，就得给它一个明确的、可自动验证的完成标准。

---

## 二、地基：让 AI 读得懂的治理配置

第一个认知是：**规范文档对 AI 无效**。AI 不会读 Wiki，它读的是系统 prompt 和结构化配置。

我们需要三样东西：

**完成标准（Definition of Done）。** 不是「把登录功能做好」这种模糊描述，而是一组可自动验证的条件。每一项绑定一条 shell 命令，exit 0 通过，非 0 失败。Agent 每次修改后自己跑一遍，全绿了就完成，有红的就继续修。

关键设计是升级条件：Agent 不是每一步都等人审批。只有两种情况停下——连续多轮自检没有改善，或者遇到了预设的需要人工介入的决策类型（如新增依赖、数据库迁移、破坏性 API 变更）。

**架构约束（Constraints）。** 把架构规则写成可执行的检查。依赖方向对不对、有没有循环依赖、API 是否向后兼容——每条规则绑一条命令。规则分两级：error 阻塞完成，warning 仅记录不阻塞。还有一个被低估的设计是**反目标**——明确告诉 AI 不要做什么，经验上这比告诉它要做什么更能防止跑偏。

**决策记录（Decisions）。** Agent 在做出架构选择时，顺手写一条结构化记录——不是什么繁重的文档流程，就是几行 YAML：做了什么选择、拒绝了什么方案、为什么、影响范围。追溯性有了，但不增加摩擦。

这三样东西合在一起，形成了 Agent 的「运行环境」——它不需要人类在旁边看着，只需要遵守约束、通过自检、记录决策。

## 三、对抗审查：另一个 AI 来挑刺，可靠吗？

一个自然的想法是：用另一个 AI Agent 来审查主 Agent 的设计——你写代码，我来挑刺。

但这里有一个隐蔽的陷阱：**LLM 的顺从偏误（Sycophancy）**。当审查 Agent 说「你这个设计在并发场景下是不是有问题」，主 Agent 倾向于说「你说得对，我改」，即使原来的方案本身没有问题。提问一方天然带有「质疑即权威」的修辞优势——这是从人类对话数据里学来的模式。

结果是方案在来回辩论中退化，收敛到更差但双方都「同意」的版本，浪费时间和算力。

**我们的解决方案不是放弃审查，而是改变审查的规则：**

第一，**能自动验证的，不辩论**。让测试和门禁说话，不需要 Agent 之间口头争对错。

第二，**必须人工判断的，用并行盲审**。多个审查 Agent 独立评审同一方案，互不知晓。只有两个以上独立指出的问题才反馈修改。单一审查者的意见只记录不阻塞——这是消除顺从偏误的关键机制。

第三，**质疑要带证据**。审查者必须提供可验证的反例——具体的、可重现的失败场景。不给具体场景的质疑不被受理。

核心原则一句话：**把「观点之争」变成「事实检验」**。

## 四、控制塔：人需要看到全局

配置解决了单个 Agent 的行为约束。但当项目数量上来后，一个新需求出现：**人需要一个地方同时看到所有项目的状态**。

我们设计的是一个轻量平台，核心原则是**异常驱动**——全绿的项目不需要你看，只有黄灯、红灯的项目才值得点进去。

**项目总览**是第一层。所有项目铺成卡片网格，每张卡片只放核心信息：健康灯、服务列表、约束合规率、Agent 数量、进度条、Token 消耗。目标是 30 秒扫完，心里有数。

**项目详情**是第二层。点击卡片进入——服务拓扑图（前端→后端→数据层，各节点带健康状态）、约束合规矩阵（每条规则的实时状态）、Agent 活动时间线、最近的决策记录。四块面板同屏，一眼看全项目内部情况。

**Live Inspection** 是第三层。点击任意 Agent，进入运行时视图——当前在做什么、自检进度（逐个约束检查的实时状态）、改了哪些文件、迭代历史。如果 Agent 跑在云上，这是你唯一能看到它「执行现场」的地方。

### 升级收件箱

当 Agent 遇到需要人类拍板的事——比如选 Kafka 还是 Redis Streams——它会触发升级事件，出现在「升级收件箱」。

关键交互设计：不要在狭窄的侧边栏里做决策。点击升级事件，右侧滑出一个完整的决策面板——背景是什么、有哪些方案（优劣并列，Agent 推荐绿色高亮）、Agent 的分析和风险提示、影响哪些服务和约束、预估成本。底部是大而清晰的决策按钮。

人不需要切到项目里读代码才能做判断。决策所需的上下文，面板本身就提供齐了。

## 五、人机交互通道：不只是看，还要能说

有了观察窗口，下一个问题是：**如果我想干预 Agent，怎么办？**

终端时代，人随时可以插话。云 Agent 时代，需要把这个能力升级为**异步消息通道**。你在平台上发一条指令，Agent 在下一轮自检时拉取并标注已读。不是实时对话，而是异步通信——你不需要在线等回复，Agent 也不会空转等人。

从干预的轻重来看，形成了三级分层：

- **Agent Console**：日常对话、轻量纠偏。比如「改用 argon2」「先做登录，注册延后」——异步消息，Agent 下轮生效。
- **升级收件箱**：重大架构决策。Agent 停下来等人拍板，提供完整上下文后你再做决策。
- **暂停/终止**：紧急操作，即时生效。

## 六、三个关键认知

回顾整个设计过程，有三个认知贯穿始终：

### 1. 非会话式 Agent 是主流，会话式是逃生舱

日常工程任务天然适合非会话式。Agent 每次迭代只需要三样东西：文件系统的当前代码、约束规则、完成标准。它不需要记住 50 轮对话历史——代码在文件里，决策在记录里。会话式的真正价值是架构探索、复杂 Debug 等需要人类深度参与的少数场景。

### 2. 能验证的不辩论

LLM 的顺从偏误是对抗审查的致命弱点。正确的做法是：审查者必须提供可验证的反例，用测试结果而非口头辩论决定对错；用并行盲审消除 Agent 之间的相互影响；分歧无法消解时果断升级人工。

### 3. 平台是云 Agent 的唯一界面，不是加分项

本地 Agent 时代，终端是人机交互的默认界面，平台是锦上添花。云 Agent 时代，平台是唯一界面。这意味着结构化遥测不是「最好有」而是「必须有」，Agent Console 不是可选功能而是核心通道。

## 七、怎么落地

我们不想一上来就建一个重平台。正确的节奏是渐进式叠加：

**第一步**，在任何一个项目里加上三份治理配置，今天就能用。我们做了一个初始化脚本，支持 Node/Python/Go/Rust/Java 五种技术栈自动适配 check 命令。

**第二步**，当并行项目超过 5 个时，加一个轻量聚合层。每个项目产出一个标准化的状态文件，一个脚本聚合所有状态渲染成面板。不是平台，就是一个脚本。

**第三步**，当项目达到几十个、Agent 类型不限于 Claude Code 时，才需要真正的控制塔平台。但此时你已经从前两步积累了足够多的经验，知道什么值得自动化、什么不值得。

---

**AI coding 规模化之后，瓶颈从「写代码」转移到了「管项目」。谁能更早建立起对 AI 交付物的治理能力，谁就能更早释放 AI 的规模化生产力。**
